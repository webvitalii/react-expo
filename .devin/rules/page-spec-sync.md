# Keep Page Specs in Sync with Code

This app documents every page **colocated with its code**: each page lives in
`src/pages/<PageName>/` with the component in `index.tsx` and its spec in
`spec/README.md`. Specs and code MUST stay in sync. Treat the spec as part of the
deliverable, not an afterthought.

This rule currently applies only to pages under `src/pages/`.

## When to update specs

Whenever you change code under `src/pages/`, update the matching spec **in the same
change set** if the change affects observable behaviour, structure, or contracts. This
includes:

- Adding, renaming, or removing a **page** → update its route file under `src/routes/`,
  the nav entry in `src/components/Navbar/navConfig.ts`, and the page's `spec/README.md`.
  (`routeTree.gen.ts` is generated by TanStack Router — never hand-edit it.)
- Changing **UI fields, flow, or state** on a page → update that page's spec UI/Flow/State
  section.
- Changing the **i18n keys** a page uses (the `useTranslation` namespace or any `t('...')`
  key) → update the locale files at `public/locales/<lang>/<namespace>.json` and the
  spec's "i18n Keys" section.
- Changing a page's **route** (path, `$lang` prefix, loader, or namespace preloading) →
  update the "Route" section of the spec and the route file under `src/routes/`.
- Changing a page's **implementation status** → update the `Status` line in the spec
  header (the spec is the source of truth for status).

## Page ↔ spec map

Derived from convention, never duplicated: for a given `<PageName>` (the PascalCase page
folder, e.g. `BmiPage`), the component is `src/pages/<PageName>/index.tsx` and its spec is
`src/pages/<PageName>/spec/README.md`. The set of specced pages is discoverable directly
from the filesystem (`src/pages/*/spec/README.md`) — there is no separate index to keep in
sync.

## Rules

1. **Ground every claim in the code.** Before writing or editing a spec's Route, UI,
   State, i18n Keys, or Files to Touch, read the referenced source (the page's
   `index.tsx` and its sibling modules, the route file under `src/routes/`,
   `src/components/Navbar/navConfig.ts`, and the relevant `public/locales/**`) and ground
   every path, field, state value, route, and key in what actually exists. Never invent
   fields, keys, or paths — if something is planned but not implemented, mark it as
   pending rather than stating it as fact.
2. **Never** land code changes that contradict a spec without updating that spec.
3. Keep the spec header `Status` line consistent with the actual implementation state.
4. When adding a new page, create `src/pages/<PageName>/index.tsx` and
   `src/pages/<PageName>/spec/README.md` by copying the canonical template at
   `.devin/templates/page-spec.md` (the single source of truth for spec structure). Add
   a `// Spec: ./spec/README.md` comment above the page's exported component. As a spec
   grows, split it into more docs inside the same `spec/` folder (e.g. `spec/state.md`,
   `spec/i18n.md`) linked from `spec/README.md`.
5. If you intentionally defer work, mark it in the spec as `Scaffold` or `Planned` and list
   the pending fields/files — do not silently leave the spec stale.
6. Prefer minimal, accurate spec edits over rewrites; preserve existing structure and links.
