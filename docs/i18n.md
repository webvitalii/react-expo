# Internationalization (i18next + i18next-cli)

Translations live in `public/locales/{lang}/{namespace}.json` and are loaded at
runtime by `i18next-http-backend`. The pipeline mirrors Tailwind's class
scanning: a CLI walks the source tree, extracts every `t(...)` / `<Trans>`
call, and **rewrites the JSON files** to contain exactly those keys — unused
keys are pruned automatically.

## File structure

```
public/locales/
  en/
    navbar.json    # top-level navigation labels
    diff.json      # /tools/diff page strings
  fr/
    navbar.json
    diff.json
src/
  i18n.ts          # i18next runtime config (namespaces, fallback, backend)
i18next.config.ts  # i18next-cli config (extraction, pruning, types)
```

## Adding a new translated string

Always pass the English text as the **inline default**. The extractor reads
it and populates `public/locales/en/{ns}.json` automatically. Secondary
languages (currently just `fr`) get an empty string for translators to fill.

```tsx
const { t } = useTranslation('diff');
return <h2>{t('result', 'Diff Result')}</h2>;
```

Then run:

```bash
npm run i18n:extract
```

The English JSON file is updated, missing keys appear empty in `fr/`, and
any obsolete keys are removed.

## Adding a new namespace

1. Create `public/locales/en/<ns>.json` and `public/locales/fr/<ns>.json` (can be empty `{}`).
2. Use it in a component: `const { t } = useTranslation('<ns>');`.
3. Load it in the consuming route's `loader` (see [Adding a new translated page](#adding-a-new-translated-page)). **Don't** add it to `ns` in `src/i18n.ts` — that array is reserved for namespaces visible on every page (currently just `navbar`).
4. Run `npm run i18n:extract` to populate keys.

## Adding a new language

1. Add the locale code to `supportedLanguages` in `src/i18n.ts`.
2. Add the same code to `locales` in `i18next.config.ts`.
3. Run `npm run i18n:extract` — the CLI creates empty JSON files under `public/locales/<lang>/`.

## Adding a new translated page

A translated page needs two things:

1. **A route file under `src/routes/$lang/...`** so the EN/FR switcher
   stays active. The `$lang` layout calls `i18n.changeLanguage(lang)` on
   mount and the navbar's language switcher resolves the new path
   automatically.

2. **A `loader` that fetches the page's namespace(s)** so the component
   never renders against a missing translation. The loader runs in
   parallel with the route chunk download, and `defaultPreload: 'intent'`
   in `src/router.tsx` makes both start on hover.

Full example — `src/routes/$lang/tools/diff.tsx`:

```tsx
import { createFileRoute } from '@tanstack/react-router';
import i18n from '@/i18n';
import DiffPage from '@/pages/DiffPage';

export const Route = createFileRoute('/$lang/tools/diff')({
  loader: () => i18n.loadNamespaces('diff'),
  component: DiffPage,
});
```

For a route that uses multiple namespaces, pass an array:
`i18n.loadNamespaces(['diff', 'common'])`. The router's
`defaultPendingComponent` shows during the load, and the component
renders synchronously with translations already in cache.

## Dynamic keys (Navbar)

The Navbar resolves labels through `t(entry.i18nKey)` where `entry.i18nKey`
is a runtime value. The extractor can't see those strings statically, so
they're whitelisted in `i18next.config.ts`:

```ts
preservePatterns: ['navbar:*'];
```

This keeps every key in `navbar.json` even though it isn't a literal `t(...)`
call. When you add a new top-level entry to `navConfig.ts`, add its key
manually to both `en/navbar.json` and `fr/navbar.json` (or just run the dev
server — i18next will show the key name as a fallback so you see what's
missing).

## Scripts

```bash
# Write JSON to match source code (the "tree-shake")
npm run i18n:extract

# CI-friendly: exit non-zero if JSON files are out of sync (no writes)
npm run i18n:check

# High-level translation completeness report
npm run i18n:status

# Generate src/types/i18next.d.ts for typed t() keys (optional)
npm run i18n:types
```

## CI integration

A husky `pre-push` hook (`.husky/pre-push`) runs `npm run i18n:check`
before every push and rejects the push if any `t('new.key')` was added
without a corresponding `npm run i18n:extract`. To bypass in an emergency:
`git push --no-verify`.

Add the same `npm run i18n:check` step to your CI pipeline as a second
line of defense.
